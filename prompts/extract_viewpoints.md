あなたはソフトウェア品質保証(QA)と形式検証のエキスパートです。
提供された「要件の構造化データ（JSON）」を分析し、Alloyで検証すべき重要な「検証観点 (Verification Viewpoints)」を提案してください。

## タスク
構造化データを読み解き、**最大8個まで**の具体的な検証観点を抽出してください。

### 観点抽出の原則
1. **具体的なスコープ**: 「デッドエンドがないこと」のような大きな観点ではなく、具体的な状態や遷移に焦点を当てる
   - ❌ 「すべての状態からの遷移が存在する」
   - ✅ 「Error状態からMaintenanceまたはSafeModeに遷移可能」
   - ✅ 「Charging状態からIdleに正常に復帰可能」

2. **優先度基準** (`extraction_perspective_and_priority.md` に基づく):
   - **Lv.1 (必須)**: 状態制御、並行・排他制御 → 形式検証に最適
   - **Lv.2 (推奨)**: 複雑な条件ロジック → 有効
   - **Lv.3 (余裕があれば)**: データ整合性制約

3. **カテゴリ分類**:
   - **Safety**: 「悪いことが起きない」（排他制御、不正遷移防止）
   - **Liveness**: 「良いことがいつか起きる」（到達可能性、復帰可能性）
   - **Consistency**: 「矛盾がない」（状態とリソースの整合性）

### 抽出すべき観点の例
1. 特定の状態からの遷移可能性（例：Error→Maintenance/SafeMode）
2. リソース排他制御（例：充電ステーションの同時利用禁止）
3. 交差点での同時進入防止
4. 状態に対するガード条件の整合性（例：バッテリー低下時の挙動）
5. 初期状態からの到達可能性
6. 最終状態への到達可能性

## 出力フォーマット (JSON)
必ず以下のJSONスキーマに従って出力を行ってください。
nameとdescriptionは日本語で出力してください。

```json
{
  "viewpoints": [
    {
      "id": "VP_001",
      "category": "Safety" | "Liveness" | "Consistency",
      "name": "具体的な観点名（例：Error状態からの復帰可能性）",
      "description": "検証内容の詳細説明（何を、どう検証するか）",
      "priority": "High" | "Medium" | "Low",
      "target_states": ["関連する状態名（任意）"],
      "target_resources": ["関連するリソース名（任意）"]
    }
  ]
}
```
