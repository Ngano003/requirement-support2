# 1-2日で可能な効果検証プラン (Quick Win Plan)

## 1. 目的
既存の要件定義書（サンプルまたは実データ）を用いて、本ツールが **「人間が見落としがちな欠陥」をどれだけの精度で検出できるか** を短期間で実証する。

## 2. 検証ステップ

### Step 1: 既知のサンプルを用いたベースライン検証 (Day 1 - 午前)
*   **対象**: `requirements/smart_lock.md` (スマートロックの要件定義書)
*   **内容**: このファイルには、意図的にいくつかの欠陥（Dead Ends、仕様漏れ）が埋め込まれている。ツールを実行し、これらを正しく指摘できるかを確認する。
*   **期待される効果**:
    *   **Dead End検出**: 「Shutdown状態から復帰する手順がない」ことを指摘できるか。
    *   **Missing Else検出**: 「解錠中にバッテリー切れになったらどうするか？」等の未定義動作を指摘できるか。

### Step 2: 実データを用いた検証 (Day 1 - 午後 〜 Day 2)
*   **対象**: 貴社の既存プロジェクトの要件定義書（Markdown, Word, Text等）
*   **内容**: 実際に運用されている（または過去に手戻りが発生した）ドキュメントをインプットとする。
*   **アクション**:
    1.  ファイルをプロジェクトフォルダに配置。
    2.  ツールを実行し、レポートを出力。
    3.  検出された欠陥が「妥当な指摘」か、エンジニアによるレビューを行う。
*   **期待される効果**:
    *   既存ドキュメントに潜む「暗黙の了解」や「記載漏れ」を洗い出す。
    *   「もしこのツールを要件定義時に使っていれば、この手戻りは防げたか？」を評価する。

## 3. 必要な準備
*   **環境**: 既に構築済みのローカル環境 (Python, LLM APIキー)。
*   **データ**: 検証対象のドキュメントファイル。

## 4. アウトプットイメージ
検証終了後、以下のようなレポートが自動生成される。

> **[Critical] Dead End Detected**
> *   **Location**: `smart_lock.md`, Section 3.3
> *   **Issue**: `Shutdown` 状態への遷移はあるが、そこからの脱出遷移（充電後の復帰など）が定義されていない。
> *   **Impact**: 電池切れ後、永久にシステムが使用不能になる可能性がある。
